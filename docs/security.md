# Security

## Overview

The system implements multiple security layers to protect sensitive telemetry data and delivery metrics.

---

## Authentication

### API Key (Frontend → Backend)

All endpoints require the `X-Api-Key` header.

| Aspect | Detail |
|--------|--------|
| Header | `X-Api-Key: <value>` |
| Alternative | `Authorization: Bearer <value>` |
| Validation | Direct comparison against `DASHBOARD_API_KEY` env var |
| Development mode | If `DASHBOARD_API_KEY` is not set, all requests pass |
| Error | HTTP 401 `{"error": "Unauthorized"}` |

**Flow:**
1. Frontend sends `X-Api-Key` on every request (configured via `VITE_API_KEY`)
2. Lambda extracts the header (case-insensitive: `x-api-key` or `X-Api-Key`)
3. Compares against `DASHBOARD_API_KEY`
4. If mismatch → 401

### IAM Authentication (Backend → RDS)

The database connection uses AWS RDS IAM authentication.

| Aspect | Detail |
|--------|--------|
| Mechanism | Token generated by `boto3 rds.generate_db_auth_token()` |
| Token duration | 15 minutes |
| Renewal | Automatic every 14 minutes (`_MAX_TOKEN_LIFETIME_SECONDS`) |
| Fallback | If `LEANMOTE_DB_PASSWORD` is defined, uses direct password |
| SSL | Enabled on all connections |

**Benefits of IAM auth:**
- No passwords in code or environment variables
- Temporary, auto-renewable tokens
- Audit trail via CloudTrail
- Centralized revocation via IAM policies

## Authorization

### Multi-tenancy by Organization

All queries filter by `id_organization`:

```sql
WHERE id_organization = %s
```

This ensures data isolation between organizations at the query level. There is no cross-org access mechanism.

### Org ID

The `org_id` is sent as a query parameter. On the frontend, it is configured via `VITE_DEFAULT_ORG_ID`.

---

## Data Protection

### Authentication Tokens

Claude Code tokens are **never stored in plain text**. The SHA-256 hash is stored:

```
authorization_token_sha256  CHAR(64)  NOT NULL
```

This allows:
- Joining with `user_identity_map` to resolve identities
- Not exposing real tokens in case of a DB breach

### Query Parameterization

All SQL queries use parameters (`%s` placeholders):

```python
cursor.execute("SELECT ... WHERE id_organization = %s AND DATE(created_at) BETWEEN %s AND %s", (org_id, start_date, end_date))
```

PyMySQL handles parameter binding and escaping, preventing SQL injection.

### Secrets

| Secret | Storage |
|--------|---------|
| DB credentials (prod) | AWS SSM Parameter Store (SecureString) |
| DB password (dev) | Local `.env` (excluded from git) |
| Dashboard API key (prod) | AWS SSM Parameter Store |
| Dashboard API key (dev) | Local `.env` |

## Network Security

### CORS

```
Access-Control-Allow-Origin: *
Access-Control-Allow-Headers: Content-Type,X-Api-Key,Authorization
Access-Control-Allow-Methods: GET,OPTIONS
```

**Note:** In production, `Access-Control-Allow-Origin` should be restricted to `https://ai-governance.leanmote.com`.

### HTTPS

- **Production:** HTTPS enforced via API Gateway
- **Development:** HTTP on localhost (acceptable for dev)

### SSL on DB

All connections to RDS use SSL:

```python
pymysql.connect(..., ssl={"ssl": {}})
```

## Logging and Auditing

### Structured Logging

Each request is logged in JSON format:

```json
{
  "level": "info",
  "timestamp": "2024-03-15T10:30:00Z",
  "type": "request",
  "status": "success",
  "request_id": "abc-123",
  "org_id": 1,
  "message": "GET /api/v1/usage"
}
```

Logs are compatible with CloudWatch Insights for advanced queries.

### Information NOT logged

- Authentication tokens
- Database passwords
- OTEL payload content

## Considerations and Future Improvements

| Area | Current State | Recommended Improvement |
|------|---------------|------------------------|
| API auth | Simple API key | OAuth 2.0 / JWT with expiration |
| CORS | Permissive (`*`) | Restrict to production domain |
| Rate limiting | Not implemented | Throttling per API key/IP |
| Org isolation | Query-level filter | Row-level security in DB |
| Audit trail | CloudWatch logs | Dedicated audit table |
| Input validation | Basic (org_id, dates) | Stricter format validation |
| Encryption at rest | RDS default | Verify KMS encryption is enabled |
